Compression (ZLIB) Settings dialog

 

 

This dialog allows you to change the settings used for compression and decompression.  The version of the ZLIB compression library used (currently 1.2.3) is displayed in the title bar.

 

Compression level

This trades off speed with amount of compression.  Note that a high value (such as 8 or 9) may not give significantly better compression but will be a lot slower.  The default value is a reasonable compromise.

Window size

Larger values give better compression at the expense of memory usage.  This value must be the same or larger during decompression as was used for compression, otherwise the decompression will fail.

Memory usage

This trades off speed with memory usage.  A high value is recommended since memory is cheap nowadays.

Data recovery (full flush) point

Full flush points allow for the recovery of some data even if part of the compressed data stream is corrupted.  The zlib algorithm can write full flush points at any time, which allows all data from that point to be analysed independently of data before it.  The disadvantage is that writing a full flush point can slow the compression (dramatically if written very often) and increase the compressed size.

 

Note that HexEdit cannot write full flush points more often than the size of its internal buffer (16Kb) so any value entered is rounded up to a multiple of the buffer size.  Also a value of zero means that no full flush points are written.

 

Generally, full flush points are not necessary unless there is a chance of corruption so the default (0) is recommended.  On the other hand very occasionally (eg, every 10 MBytes) writing them does not add significantly to the size or speed of compression.

Header

ZLIB compression provides a small header that allows it to keep track of things and do error checking etc.  The gzip utility (commonly used on Unix) uses exactly the ZLIB compression algorithm but needs to store a little more information so uses its own header.  When decompressing ZLIB can detect header and handle both.

 

For absolute minimum compressed size you can turn off the ZLIB header altogether.  However, this is not recommended since it turns off a lot of error checking.

Strategy

Various strategies are provided to handle different sorts of data.  When used for general purpose compression the default strategy is recommended.  You might like to experiment with specific strategies if you always compress the same sort of data.  Please refer to the ZLIB documentation for more information.

 

Note that decompression does not need to specify a strategy.  That is, the compressed data can always be decompressed irrespective of the strategy used for compression.

Note:

There is no option to set the compression algorithm, as the ZLIB library currently (version 1.2.3) only supports one algorithm (Z_DEFLATE).
