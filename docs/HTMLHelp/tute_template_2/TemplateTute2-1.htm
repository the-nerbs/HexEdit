Step 1: The Data File Format

 

     

 

The first requirement for creating a template is that you have some idea of the data file format, whether you are working from a rough description or a formal specification (or perhaps you even designed the format yourself).

 

The most difficult situation is if you are trying to decipher a format about which you have no information.  You will need to create lots of different files to try to work out the differences, but mainly have lots of patience.  We will not cover this in the tutorial as whole books could be written about it.

 

HEG File Specification

 

To best demonstrate several template features we have created a simple, fictitious file format for a simple 2-D graphics program.  We will give files of this format the extension .HEG (for HexEdit Graphics).  There is an example file of this format (in the folder where HexEdit is installed) called TUTORIAL.HEG.  Let's open it now.

 

1. First make sure template editing mode is on by opening the Display page of the Options dialog.  This step is no longer necessary in HexEdit 3.2.

 

 

              Figure 1.1 Turning On Template Editing in the Display Options page.

 

2. Open the "TUTORIAL.HEG" file (stored in the directory where you installed HexEdit).

 

If you forget where you installed HexEdit right-click the HexEdit icon and Select Properties.  Then find the "Target" field on the Shortcut page of the Properties dialog.  If there is no Shortcut page then you are looking at the properties for the actual EXE file, just find the "Location" field on the General page.

 

The HEG format simply consists of a header followed by zero or more graphical objects, and finally a color table.

 

The description "header" is often the name used for a set of global values that occur before the bulk of the data.

 

The first noteable feature of this format is the color table at the end of file.  Having this sort of table last is common where information is gathered while a file is being written and "header" type information is written after the data.  It is often called a "trailer" rather than header for obvious reasons.

 

For our format putting the color table last is a bit contrived since the color information is available to be written at the start of the file.  It is used to demonstrate later in this tutorial how to process data out of order, specifically to look up information from the color table during processing of the graphical objects.

 

The other noteable feature is that the fictitious graphics program supports grouping of graphical object into combined objects.  Although it only supports two primitive graphical objects (rectangle and circle) grouping allows nested objects to be created by the (fictitious) user to any depth.  To reflect this type of data we need to use a recursive data structure.

 

Header

 

The header is 16 bytes long consisting of four 4-byte fields: a "magic" number, the date the file was originally created, the date the file was last modified, and the address of the color table.

A magic number is a value that is always the same for a particular type of file.  Magic numbers are usually 32 bit numbers stored in the first 4 bytes of a file.  They're simply used as a double check that this is actually the type of file we expect.  We will validate this field in the template to make sure it contains the correct value.  Sometimes, the magic number is the birthday of the designer of the format.

 

The date fields are not really necessary as this information can usually be obtained from the filesystem (stored in the directory entry for the file).  However, we will use them to demonstrate a bit of date arithmetic later.  The format is a "normal" time_t - see Date and Time Formats for more information.

 

The color table address is stored at the start of file so we can access it without having to scan all the graphical objects in between.  When a file of this type is (fictitiously) written out the address would initially be unknown and just written as zero then after the graphical objects and the color table have been written the software would jump back to byte 12 of the file and fill in this value.

 

Note that the color table address is only 32 bits so the file length is effectively limited to 2^32 bytes or 4 Gbytes.  Most file formats will store absolute file addresses in 32 bit numbers since, until fairly recently, filesystems (and operating systems) only supported 32 bit file sizes.  (In fact, for convenience, file sizes are usually stored in *signed* 32 bit numbers so many files are actually limited to 2 Gbytes.)

 

Graphical Objects

 

There are 3 types of graphical objects: a rectangle, a circle, and a "grouped" object containing one or more other objects.  The primitive objects (rect and circle) have a color value which is simply an index into the color table.  All objects have coordinates (X and Y) which stores their position which is absolute for an "ungrouped" object or relative to the containing object if they are part of a grouped object.

 

Each object begins with a byte which identifies what type of object it is.  This is 1 for a rect, 2 for a circle or 10 for a "grouped" object.  A type of zero is used to signal the end of the list of objects.

Color Table

 

The color table consists of the size of the table followed by the corresponding number of entries.  Each entry has a 13 byte name plus a single byte red, green and blue value.  The name is a null-terminated ASCII string (unless it is 13 characters exactly).
