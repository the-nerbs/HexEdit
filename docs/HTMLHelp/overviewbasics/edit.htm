Editing

 

When editing a file HexEdit behaves much like a word-processor or editor. You can move the cursor around using the keyboard or the mouse (see Moving Around). You can select text with the keyboard or mouse and cut, paste etc (see Selecting and the Clipboard).

 

One difference is that there are two separate areas for editing – the hex area and the character area (although either area can be disabled). You can enter any byte value in the hex area but only by typing two hex digits – HexEdit beeps if you type any other characters. On the other hand you can only enter text charcaters (ASCII or EBCDIC) in the character area.

 

To quickly switch between the areas use the Tab key.

 

Modes

 

There are two sets of toggled states that affect editing behaviour.  The first toggles between read-only and read-write modes; the 2nd toggles between overtype and insert modes.

 

If you are in read-only mode then you cannot modify the file at all. This mode is separately controllable for each window so you can have two windows on the same file in different modes. But if the file itself is opened read-only then all windows for the file must remain read-only. The RO/RW indicator on the status bar indicates the current mode – you can double-click this indicator to toggle the state.

 

 

The Insert/Overtype mode is changed (similarly to the read-only mode) by double clicking the INS/OVR status bar indicator toggles the state. Both modes have uses and limitations as explained below..

 

In insert mode the length of the file may change. As you enter bytes the following ones are shifted forward. Similarly deleting, cutting etc removes data and shifts the following bytes backward. But in overtype mode existing bytes are never moved, instead changing bytes destroys their previous value. Further if you try to delete or cut to the clipboard in overtype mode HexEdit asks if you want to switch to insert mode.  Pasting in overtype mode asks if you want to overwrite or switch to insert mode.

 

In many types of binary files it is essential that any changes do not move bytes around and the length of the file does not change – for safety always edit these types of files in overtype mode.  Another problem with insert mode is that if you insert or deletes bytes near the start of a very large file then the whole file needs to be rewritten which may take a lot of time.

 

On the other hand it is often very useful to be able to insert bytes within a binary file. Also, to add to the end of a file you must be in insert mode.

 

Undo

 

When you make changes to a file, HexEdit keeps track of these changes and allows you to undo them. In fact HexEdit uses this undo information plus the original file on disk to show you what the file is currently like. This means that you can quickly edit very large files since, unlike many editors, the whole file does not have to be read into memory (or some sort of virtual memory). But this system does have the disadvantage that you cannot undo changes that were made before you last saved the file to disk.  Also as you make more and more changes the display takes longer and longer to regenerate -- but this is not normally a problem unless you run a macro or replace operation that makes lots of changes.

 

As well as a file-level undo, which keeps track of changes to the file, HexEdit also has a window-level undo that keeps track of cursor movement, changes made to the display etc. However, since changes made to the file can affect the display in another window for the same file HexEdit must take special precautions to keep its internal structures consistent. If you undo a change to the file in one window, all display changes in other windows on that file are undone back to the point in time when the original file change was made. Moreover, if you try to undo back past a change made in another window HexEdit will ask you if you really want to undo the change.

 

Change Tracking

 

HexEdit can show all the changes you have made to a file, since you opened (or last saved) it.  It shows where bytes have been replaced, inserted and deleted.  Note that, it does not show how the changes were made - for example, if you delete 20 bytes then insert 20 bytes at the same location then this will be shown as 20 bytes having been replaced, since that is the effective difference with what is stored on disk.

 

Replacements are shown with a red underline, insertions with a (lighter shade of) red background, and deletions are marked with a red block at the deletion points.  In the hex area the deletion also shows a count of the number of bytes deleted when it is less than 10.  The change tracking color can be changed from the default of red by changing it in the current color scheme using the Color Options page of the Options dialog.

Display of change tracking can be toggled using the button on the Edit Bar, or using the check boxes in the Document Edit Options page of the Options dialog.

 

File Handling

 

HexEdit behaves like many Windows programs, such as word-processors, when it comes to the opening, closing, saving etc of files. The only difference is that instead of HexEdit being associated with a particular type of “document”, you can work with any type of file.  In the File Open Dialog you will see (by default) all files, since the default file type is “All Files (*.*)”.  You can, however, select different file filters from the "Files of Type" drop down list -- you can even control what filters are shown in the filter list with the Filters page of the Options dialog.

 

The Read only check box in the File Open dialog allows you to open a file for read only. You may want to do this so that you don’t accidentally modify the file.  If the file must be opened read-only because it is a hidden or system file or because something else is reading it then HexEdit will warn you. Note that even if a file is not opened read-only a window on that file may be read-only (see above).

 

Normally HexEdit likes to prevent a file from being modified while it has the file open.  This is because HexEdit does not cache the file in memory but will update the display directly from the disk file.  However, sometimes you may want to allow other programs to access the file at the same time which is when you should use the Shareable check box.  However, not that in this case if something else modifies the file contents may change unexpectedly.
